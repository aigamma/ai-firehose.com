Collective communication is the plumbing that holds distributed training together. Once a model is split across many accelerators, the devices must constantly exchange tensors, averaging gradients, sharing weights, redistributing activations, and doing so with operations that involve a whole group of devices at once rather than a single send between two. These group operations are called collectives, and a small standard vocabulary of them, implemented in libraries optimized for the hardware, underlies essentially every form of parallel training.

The core primitives each have a precise meaning. All-reduce combines a tensor held on every device, by summing or averaging, and returns the same combined result to all of them, the operation that averages gradients in data parallelism so every replica applies the same update; all-gather collects each device's distinct shard and assembles the full tensor on every device, how fully sharded data parallel reconstructs a layer's complete weights from the shards just before using them; reduce-scatter does the reduction but leaves each device holding only its slice, the counterpart used to distribute averaged gradients back to the shards; broadcast copies one device's tensor to all; and all-to-all has every device exchange a distinct piece with every other, the pattern expert parallelism uses to route tokens to the devices holding their experts.

These collectives matter to performance because communication, not computation, is what most often caps how far training scales. Every collective moves large tensors across the links between devices, so its time is governed by interconnect bandwidth and latency, and a faster fabric within a node with slower links between nodes makes the topology of where data must travel a first-order concern. As a cluster grows, the communication volume can come to dominate, so the central engineering goal is to overlap collectives with computation, hiding the data movement behind useful work, and to keep tensors on the fastest available links.

Each parallelism strategy is in effect a particular pattern of collectives. Data parallelism is an all-reduce of gradients each step; tensor parallelism inserts all-reduces inside a layer to stitch together partial results from a split matrix multiply; fully sharded data parallel is built from all-gather and reduce-scatter; pipeline parallelism mostly uses point-to-point sends to pass activations between stages. Understanding which collectives a configuration relies on, and how heavily, is how one predicts whether a training setup will scale or choke on its own communication.
