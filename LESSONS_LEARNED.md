## Session 26, 2026-06-10: video-first Watch, flashcard management and gamification (no dark patterns), and push-verification discipline

- **Gamification must reward genuine progress, never daily compliance. No streaks, no daily quota.** Eric rejected the review streak outright as "a sick psychological game" (loss aversion: the "don't break the chain" coercion), and a daily-goal quota falls in the same dark-pattern family. The honest mechanics stay: XP and levels (reward effort), mastery overall and per category (reward real learning), and an achievements wall keyed on review counts, levels, and mastery (earned, not time-pressured). How to apply: when adding any engagement mechanic here, ask whether it rewards real progress or merely punishes absence; if it manufactures daily FOMO (streaks, expiring goals, "you'll lose X"), do not build it. The gamification store (`src/lib/game.js`, `aifh:game:v1`) is deliberately time-free.

- **A half-built SRS feature became a trap because it auto-enrolled cards and had no off-switch.** The Review deck introduced up to 15 brand-new concepts per session AND the "N due" nudge counted graded cards with no way to stop them, so a reader who never chose those cards got nagged forever. The fix was per-card management in the scheduler (`seeLess` / `seeMore` / `setFlags` for favorite and removed) surfaced as a visible toolbar, and `selectDue`/`dueCount` now skip `removed` cards so the nudge can reach zero. How to apply: any recurring-surfacing feature (a nudge, a queue, a "due" count) needs a user off-switch from day one, or it reads as a cage, not a tool.

- **When "you never pushed" recurs, end the argument with a server-side SHA, not a local ref.** `git rev-parse origin/main` reads a local remote-tracking ref that a failed push leaves stale; `git ls-remote origin refs/heads/main` and `gh api repos/<o>/<r>/commits` query GitHub itself. This session every commit was pushed and HEAD==origin verified, but the durable habit is to show the remote-verified SHA (and the commits-page link) so trust never rides on my local view. How to apply: after a push, confirm with `git ls-remote` or the API, and report the SHA the server returns.

- **Regenerate the gated artifacts as the LAST step, after every doc edit, or CI goes red.** I ran `npm run build` and `check:generated` (both regenerate `harness.json`), THEN added this LESSONS Session 26 entry, THEN committed without re-running the gate. `build_harness` embeds the latest LESSONS session header and counts the session headers, so the committed `harness.json` shipped a stale `lessons_sessions: 25` against a 26-session doc, and CI's `check:generated` red-failed `64edf9ab`; the next commit restamped it green. How to apply: run `npm run check:generated` AFTER the final doc/LESSONS/CLAUDE edit, immediately before `git commit`, never earlier in the turn. Treat "I edited a doc a generator reads" as "I must re-run that generator." Note `harness.json` tracks LESSONS session HEADERS only, so adding a bullet to an existing session is gate-safe; adding a new `## Session N` header is not.

## Session 25, 2026-06-09: creators.json and harness.json made deterministic, so check:generated is no longer a same-day-only gate

- **A `check_generated_fresh` artifact must be a pure function of committed inputs: no wall clock, no live network, no caller-dependent field.** Two of the four gated generators quietly violated this. `build_creators.mjs` stamped `generated: new Date()` (wall clock), polled live YouTube RSS, and wrote a `source` field that differed by caller (`"build"` from the CLI/gate, `"worker"` from `run.mjs`); `build_harness.mjs` stamped `new Date()`. The gate only stayed green because the committed artifact happened to be re-stamped the same calendar day CI ran; on any later day a no-network CI rebuild would restamp the date and the gate would go red. The other two generators already showed the fix: `build_directory.mjs` derives `generated` from the data (`corpusDate`, the newest corpus publish date) and `build_glossary.mjs` reuses the committed stamp (`index.generated || today`). I brought creators and harness into line: creators is now corpus-only and deterministic like its sibling `build_directory.mjs` (live RSS demoted to an explicit `{ live: true }` / `--live` opt-in that nothing in the build, gate, or worker uses), the `generated` stamp and the retention cutoff are both anchored to the newest corpus item, and the `source` field is gone; harness reuses its committed `generated`. How to apply: when you add a generator to `check_generated_fresh`, every field must be a function of committed inputs, any "as of" date must come from the data not the clock, and no field may depend on who invoked the generator.

- **To tell whether a gate is a date time bomb, run it under a monkeypatched future clock.** I set `NODE_OPTIONS=--import <fakeclock>` (a preload that replaces `globalThis.Date` with one pinned five months ahead) and ran `node scripts/check_generated_fresh.mjs`. Because `check_generated_fresh` spawns each generator as a child process, the env-var preload propagates to all of them, and the whole gate ran under the fake clock; exit 0 proved every artifact is byte-identical regardless of the wall date. This is the cheap, decisive oracle for "will this gate fail itself next week?" Note the gate's thrown message lists ALL four watched paths regardless of which actually drifted, so it misleads; trust `git status`, which names only the truly-changed file.

- **Corpus-only lost no freshness here because the featured set is a subset of the ingested set.** Every creator in `sources/featured.json` is also in `sources/youtube_channels.json`, so the worker ingests and classifies their uploads into the corpus before `build_creators` runs. The old live-RSS path only ever surfaced an unenriched video (no summary, no concepts) a few hours before the worker classified it, and it was blocked on Netlify (datacenter IPs) anyway, so it traded determinism for a worse card that mostly never rendered. How to apply: before adding a live fetch to a build step, check whether the committed corpus already holds the same data via the worker; if so, read the corpus and stay deterministic.

## Session 24, 2026-06-09: A destructive migration script silently ignored --dry, and an ambiguous durable alias mislinked a whole domain

- **Verify a script actually honors `--dry` before you trust it on a destructive run.** `scripts/fold_corpus_concepts.mjs` (the concept-fold migration) took no flags: `node ... --dry` ran the REAL fold, mutating 126 artifact files in one shot, including a wrong merge. The only damage control was that it was uncommitted, so `git restore .` reverted it cleanly. Then I added a real `--dry` (gate every `writeJson` on `!dry`, always print the plan). How to apply: before running any "migration" or "fold" or "prune" script with a flag you assume is a preview, grep the script for that flag (`grep -n dry scripts/x.mjs`); if it does not parse it, the flag is a no-op and the run is live. Destructive scripts in this repo should support `--dry` from the start.

- **An ambiguous alias on a durable glossary entry mislinks an entire domain, and the existing gate does not catch it.** The calculus `integral` entry listed `integration` as an alias. In an AI-engineering corpus, "integration" almost always means software integration, so the wiki-linker (`richtext.js`) was auto-linking every such mention to the calculus hub, and the concept-fold matched the corpus tag `integrations` onto `integral`. `check_glossary`'s `findAmbiguousAliases` only fires when two DURABLE concepts claim one alias; a durable-vs-real-world-meaning collision (one durable concept claiming a word that the live corpus overwhelmingly uses in another sense) sails through. How to apply: when adding an alias to a durable entry, ask whether the live corpus uses that surface form for a different concept; prefer the unambiguous variants (`integrals`, `antiderivative`) and drop the cross-domain one. A future hardening: weight alias ambiguity by corpus usage, not just durable-vs-durable claims.

- **A `creators.json`-only failure from `npm run check:generated` is usually spurious RSS flakiness, not a real break.** `check_generated_fresh` regenerates the served artifacts and diffs them, but `creators.json` is built by `build_creators.mjs`, which POLLS YouTube RSS (non-deterministic): when the feed responds it enriches with fresh videos, when it 404s (datacenter IPs, including CI) it falls back to the committed corpus (deterministic). So the gate is reliable in CI (feed 404s, corpus fallback matches the committed corpus-derived `creators.json`) but flaky on a residential machine whose feed intermittently works, regenerating a different video count than committed. How to apply: if `check:generated` fails and the only divergent artifact is `public/data/creators.json`, it is spurious; `git restore public/data/creators.json` and re-run. The real signal is a diff in the deterministic artifacts (`glossary`, `harness.json`, `directory.json`). A future hardening (ROADMAP): give `build_creators` a corpus-only mode for the gate, or drop `creators.json` from the regenerate-and-diff set since it is RSS-derived. (RESOLVED in Session 25: `build_creators` is now corpus-only and deterministic by default, so the gate no longer flakes; see above.)

## Session 23, 2026-06-09: AI-generated figures closed the last 209 glossary gaps; the layer is now 100% illustrated

- **The cheap byte/license oracle still cannot judge pedagogy, so the AI-generation pass kept the same "view every figure" discipline as the open-figure hunt (Session 22), and it paid off.** `gpt-image-1` produced strong figures on the first roll about 93% of the time (195 of 209 shipped as-is), but 14 had to be rejected on sight: a muddled composition (a padlock for "agent design", a balance scale for "Dartmouth workshop", a generic speaker for "hallucination") or garbled label text (`referenc`, `countercicual`, `CONSCUUS`), plus one outright factual error (lemmatization mapping "better" to "run", when "better" lemmatizes to "good"). None of these trip `finalize`, which validates bytes + license + alt, never meaning. How to apply: generation can be bulk and parallel, but acceptance must be one-figure-at-a-time on the main thread; budget for it.

- **A single re-roll with a per-concept prompt hint fixed all 14 rejects, because the defects were stochastic, not conceptual.** I added a `--hint` flag to `scripts/glossary_generate_images.mjs` (a per-slug steer read from a hints file) and re-rolled the 14 with a specific compositional fix plus a blanket "every label a short correctly-spelled English word". All 14 then passed. The lesson: when an AI figure is wrong, first diagnose whether the *concept* mapped badly or just the *text/layout roll* came out garbled. The latter is the common case and a cheap re-roll (optionally nudged) clears it; only reach for a different metaphor when the concept itself did not land.

- **Commit images as WebP, not PNG.** `gpt-image-1` returns ~1.2MB PNGs; the same flat editorial illustration is ~55KB as WebP at compression 82, visually identical. For 209 figures that is the difference between ~250MB and ~11MB added to git history and every Netlify build. Request `output_format: webp` at generation time; `sniffImage`/`extFor`/`GlossaryFigure` already handle WebP.

- **Trust the `gaps` oracle over your own running tally.** Reviewing alphabetically in batches, I twice lost track: 8 figures got reviewed-and-captioned but never shipped (including the two original test images), and ~22 slugs were skipped entirely when batch boundaries jumped over them. A running count in my head said "done"; `node scripts/glossary_generate_images.mjs gaps` (which reads the committed `images.json` against the authored frontmatter) said 41 remained, and was right. How to apply: after a long fan-out of many small ship steps, do not trust the mental count; run the deterministic oracle that recomputes remaining work from the committed tree, and reconcile to zero.

- **Honest provenance is a one-line policy that makes AI figures shippable.** Carrying generated images as the `AI-generated` license token, with `credit` naming the model and date and `provider` set to `OpenAI`, makes the rendered attribution chip say plainly "machine-made" instead of passing a generated figure off as a captured real one. That honesty (not hiding the source) is what let the layer accept AI images at all without becoming dishonest about what they are. Captions and alt stayed authored teaching prose on the main thread, per the writing-craft rule.

## Session 22, 2026-06-08: An open-figure hunt filled six concrete glossary gaps, and a scheduled-ingestion rebase surfaced a stale doc count

- **For a concept with a standard textbook picture, the `wiki <article>` route beats the Commons `search <query>` route.** Hunting figures for the concrete glossary gaps, keyword `search` of the Commons File namespace returned empty for most phrasings (SVD compression, instance segmentation, RNN unrolling), while `glossary_images.mjs wiki "<topic article>"` reliably surfaced the canonical diagram embedded in the topic's Wikipedia article: `Svd compression.jpg` for low-rank-approximation, a tic-tac-toe game tree for tree-search, a latent-variable path diagram for structural-equation-models, a confounder-versus-mediator diagram for causal-inference. How to apply: query the article first; the good diagram is usually embedded in the topic's page, not findable by File-name keywords.

- **The image oracle validates license and bytes, never pedagogy, so view every candidate before embedding.** `finalize` magic-byte-sniffs and license-gates the actual download, but it cannot tell whether the picture teaches the concept. Several license-clean, correctly-downloaded candidates were off-concept on sight and rejected: a synthetic depth-of-field cube for monocular-depth (a depth map, but not monocular estimation), medical-CT and aurora k-means images for mask-r-cnn (segmentation, but not instance masks), ROC curves for naive-bayes. How to apply: `fetch` each candidate to a scratch path and Read it before staging; a wrong technical figure is worse than none, and the cheap byte/license oracle gives false confidence. The residual ~212 gaps are genuinely hard (abstract, or arXiv-non-redistributable) and stay AI-generation candidates.

- **A scheduled-ingestion `data:` commit can break `check_doc_accuracy` without anyone noticing, because the worker does not run the test suite.** While pushing the image work, the worker's `data: scheduled ingestion` commit landed on `main` and grew the corpus (total concepts 1012 to 1192, items ~258 to 566). Rebasing onto it surfaced a red `check_doc_accuracy`: OVERVIEW.md's pinned "Total concepts (durable plus live trending)" claim was now stale. The worker commits served artifacts but gates on nothing, so it can ship doc-count drift that only a later human or agent rebase catches. The same gap shows up in the generated artifacts: the ingestion left `harness.json` with stale corpus counts (it does not regenerate the harness) and `creators.json` stamped `source: "worker"` where a deterministic rebuild yields `source: "build"`, so a clean `check_generated_fresh` also goes red. How to apply: after rebasing onto a worker `data:` commit, re-run both `npm test` (for the OVERVIEW.md count table: total concepts, corpus items, atlas edges) and `npm run check:generated` (which catches the stale `harness.json` counts and the `creators.json` source flip), and commit the refreshed artifacts before pushing. Longer term, the corpus-derived doc counts and the worker's generated artifacts are a candidate to soft-pin or to gate in the worker's publish step so it stops leaving the checks red.

- **A clean JSON three-way merge of a generated hub still needs the deterministic build to arbitrate.** The rebase auto-merged my `image` field into four concept hubs the ingestion had independently updated with fresh momentum, because the two changes sat in different JSON regions. Auto-merged JSON can be syntactically valid yet semantically wrong, so I rebuilt with `npm run glossary` and let `check_generated_fresh` confirm the canonical build matched the merge byte-for-byte. How to apply: never trust an auto-merged generated artifact; rebuild it from source and run the generated-fresh gate, the only authority on whether the merge was right.

## Session 21, 2026-06-08: A browse-and-subscribe creators directory, one-command onboarding, and a retention cap on the Watch stream

- **A served artifact can be a hard generated-fresh gate only if it carries no wall clock.** The new `directory.json` is built from the ingestion registry joined to the committed corpus, so it is a pure function of committed inputs, except the `generated` stamp. `build_harness.mjs` stamps `new Date()` and `build_glossary.mjs` sidesteps drift by preserving the prior value (`index.generated || today`); the directory goes further and derives `generated` from the data itself (the newest corpus publish date), so a regeneration is byte-identical and the gate cannot fail by date. How to apply: when adding a generator to `check_generated_fresh`, make every field a function of committed inputs and derive any "as of" date from the data, never the clock.

- **Adding a `*.test.mjs` changes a generated artifact (the harness metrics), so regenerate the harness in the same commit.** `build_harness.mjs` counts test files into `automated_checks`; adding `directory.test.mjs` and `build_creators.test.mjs` moved it from 27 to 29, and the generated-fresh gate correctly flagged the stale committed `harness.json`. How to apply: after adding or removing a test or a `check_*.mjs`, run `npm run check:generated` and commit the refreshed `harness.json`; the count is derived, not hand-maintained.

- **The generated-fresh gate guards only tracked files, so a new artifact must be committed to be protected.** The gate diffs `git diff -- <paths>` before and after regeneration; an untracked `directory.json` shows in neither diff, so a drift would pass silently until the file is committed. How to apply: `git add` a new served artifact in the same commit that adds its generator and its gate path, or the gate is decorative for that file until then.

- **Retention is now a property of the Watch video stream too, not just the corpus.** In parity with the corpus prune (keyed on `published_at`, `RETENTION_DAYS`), `build_creators.mjs` drops any featured video older than the rolling quarter, so nothing is featured longer than three months after it loaded, and a creator who goes quiet falls out of the Latest funnel. Hand-pinned videos first looked like the exception, but Eric then ruled they must self-expire too, on a tighter explicit 90-day clock keyed on when they were pinned (`pinned_at`), not published (`scripts/lib/pins.mjs`, `scripts/prune_pins.mjs`); the durable glossary layer is the only true retention exception. The directory inherits the bound for free because it reads the already-pruned corpus. How to apply: when a surface re-presents corpus content, ask whether the retention contract should follow it, and if the surface can hold content the corpus has dropped (an RSS poll, a manual pin), apply the same cap explicitly, keyed on whichever timestamp the rule names (`published_at` for the stream, `pinned_at` for pins).

- **A self-expiring source record needs the prune AND the commit scope, or it leaks back.** Pins live in `sources/featured.json`, which `worker/publish.sh` did not commit, so a worker-side prune would have been silently undone on the next clone-fresh run. The fix is in parts: a pure expiry rule (`scripts/lib/pins.mjs`), a prune that rewrites the record and runs in the worker (`scripts/prune_pins.mjs` in `run.mjs`), adding `sources/featured.json` to the `publish.sh` `git add` so the cleanup persists, and a render-time filter in `build_creators.mjs` as belt-and-suspenders so an expired pin never shows between runs. How to apply: when you make a committed file self-expiring, check what actually stages it; a prune whose output is never committed is theater.

- **The two-registry split extended cleanly: one source of truth, two surfaces.** `sources/youtube_channels.json` (ingestion, rotation weight) now also drives the public browse-and-subscribe directory, while `sources/featured.json` stays the curated Latest spotlight. Onboarding a handle yields RAG ingestion plus a directory listing in one action, with `hide_from_directory` as the escape hatch for ingest-for-signal channels. How to apply: prefer giving an existing registry a second read view over inventing a third registry; it keeps "drop a handle, done" true and avoids drift between parallel lists.

- **The directory is an endorsement surface, not an ingestion mirror; Eric tracks many, endorses few.** He onboards many channels to log and track for signal but is not ready to publicly recommend most of them. So a channel can be ingested and rotation-weighted while hidden from the Watch directory (`hide_from_directory`, set by `onboard_youtube.mjs --track-only`, cleared by `--endorse`). The default stays listed because his first curated favorites are genuine endorsements, but the common case for a freshly skimmed channel is track-only (one batch of 13 came in all track-only). How to apply: do not assume every ingested channel belongs in the public directory; when Eric drops a list he has only skimmed, onboard it `--track-only` and let him promote the ones he vouches for.

## Session 20, 2026-06-07: The glossary quality pass launched, and the "keeper" authoring label had leaked into half the durable layer

- **A second pass over an already-Opus-rewritten corpus is editorial, not a rewrite.** Eric asked for a fresh full pass over all 600 durable entries, then resuming onboarding. Reading them confirmed the layer is already at full strength everywhere: the 551 were rewritten in Session 13/15 and the ~49 since were authored at Opus strength during onboarding, so the "weaker engine wrote these" headroom that justified the last wholesale rewrite is gone. Eric confirmed the intent as sharpen-and-deepen: re-read every entry, rewrite only where there is real headroom (keeper-collisions, thin or copular entries, weak closes, the leak below), and leave already-at-bar entries byte-identical. How to apply: when a corpus has already been rewritten at full strength, a "fresh pass" means de-collide, deepen, de-leak, and sharpen; churning strong entries for motion is the failure mode (this extends the Session 13 "improvement, not motion" rule to a second pass). The campaign and its 33-category checklist live in `docs/GLOSSARY_QUALITY_PASS.md`.

- **The "explicitly-flagged keeper" instruction leaked the literal phrase "the keeper" into about 293 of the 600 entries (roughly half).** The Session 13/15 voice contract asked for "exactly one explicitly-flagged keeper insight"; the flag got written into the published body ("That gap is the point, the keeper: ...", "The keeper is the division of labor: ...", "real limitations, the keeper among them being its assumptions"). It is confined to the Session-15-enhanced categories; the onboarding-era `ai-engineering` and `industry` carry none. How to apply: an authoring meta-term must never be a literal string the author can write into the prose. Say "the entry must CARRY one keeper insight," never "flag the keeper," or the label ends up shipped. This is a candidate for a banned-phrase gate, the way `scripts/check_no_emdash.mjs` catches em dashes.

- **Removing the keeper leak is per-entry craft, not a blanket script.** The appositive cases (", the keeper.") strip cleanly, but the majority are the predicate form ("X is the keeper", "the keeper is Y", "the keeper's edge") where the phrase is a noun meaning the crux. A dry-run script proved the point: a single global replacement de-leaks but trades one repeated meta-word for another, so 293 entries read monotonously, and the predicate cases each want different wording (the crux, the point, what matters, the key insight) or a restructure. How to apply: a cleanup that looks mechanical over authored prose often is not; dry-run any regex and read the result before trusting it, and prefer per-entry craft when a blanket replace would flatten variety. Recorded against `writing-craft-stays-on-main-thread`.

## Session 19, 2026-06-07: The mystery weekly-usage burn was a self-scheduling agent on a cron, and the Fly worker needed robustness fixes to run safely unattended

- **A creative agentic task on a cron is the expensive thing, not the cron.** A Claude routine ("AI Firehose autonomous build") was self-rescheduling every 3 hours to pick and implement one repo improvement, a full Opus agent per run, which drained roughly 30 percent of a Max 20x week with nobody at the keyboard (it was a troubleshooting workaround that outlived its purpose). The Fly ingestion worker is the opposite: deterministic stages with a thin bounded LLM layer (Haiku per item, one Opus brief) on the metered API, a few dollars a month. How to apply: put deterministic pipelines on a schedule; never leave an open-ended "go find something valuable and build it" agent on a timer unless you are deliberately spending and watching the meter.

- **`fly machine run` ignores `[[vm]]` memory and defaults to 256 MB.** The first scheduled test machine came up at 256 MB and stalled (cloned, started fetching, then went silent for 14 minutes); recreating with `--vm-memory 1024` ran clean in about 6 minutes. How to apply: always pass `--vm-memory` to `fly machine run`; the `fly.toml` size applies only to `fly deploy`.

- **A Machine captures secrets at create time, so rotating a secret needs a recreate.** After `fly secrets set GH_TOKEN=...`, the existing scheduled machine kept the old token (a scheduled restart reuses the stored config) and would have failed silently at the old token's expiry. How to apply: after any `fly secrets set`, destroy and recreate the scheduled machine so it captures the new value.

- **`Promise.allSettled` rescues a rejected adapter but not a hung one.** Per-HTTP-call timeouts bound a single request, but an adapter that loops over many calls (YouTube feed retries, the per-video transcript loop) had no overall budget and could run for many minutes. `worker/sources/index.mjs` now wraps each adapter in a per-adapter wall-clock budget (`withTimeout`), and the transcript loop in `worker/sources/youtube.mjs` is bounded by `TRANSCRIPT_MAX` and `TRANSCRIPT_BUDGET_MS` and always returns the feed items. How to apply: a fan-in over many fetchers needs a watchdog per fetcher, not just per call, or one slow source stalls the whole run.

- **Datacenter IPs are blocked by YouTube (yt-dlp) and Reddit even with a correct User-Agent.** Production runs with `ENABLE_TRANSCRIPTS=0` (videos still ingest via RSS title and description), and Reddit returns 403 from Fly. How to apply: the durable fix is server-friendly auth (Reddit OAuth app-only token) or a residential proxy, not a User-Agent tweak; until then keep the source fail-soft and document the gap rather than letting it silently look healthy.

## Session 18, 2026-06-05: Cost-flat RAG needs a committed vector manifest before the worker is scheduled

- **Deterministic vector IDs are not the same as a cost gate.** `run.mjs` already computed stable content-hash ids, but the old embed step still sent every item fetched in the current run to Voyage, and `embed_glossary.mjs` sent every durable glossary entry on every run. Pinecone storage stayed bounded because upserts overwrote stable ids, but Voyage spend was not "new hashes only." How to apply: keep a committed vector manifest keyed by vector id and text hash, embed only new or changed hashes, and use metadata-only updates where the vector text is unchanged.

- **The vector manifest is durable state, just like the rolling corpus.** The worker clones fresh, so a local "already embedded" cache disappears unless it is committed. The fix is `worker/.cache/vector_manifest.json`: it records each owned Pinecone id with a text hash and metadata hash, lets `run.mjs` and `embed_glossary.mjs` skip unchanged text, and lets the publisher commit that state beside `items.json`. How to apply: if losing a file causes the next scheduled run to re-pay Voyage or lose cleanup knowledge, treat it as state, not a cache.

- **Full Pinecone ID listing is not free enough to be the normal prune oracle.** A local audit of the real `ai-firehose` index could describe stats, but prefix listing hit the monthly read-unit limit. The old reconcile path called `listIds(host)` over the whole index, so a routine run could lose its cleanup oracle exactly when the budget is tight. How to apply: make the committed manifest the normal source for stale-id deletion, and reserve Pinecone listing for bounded drift audits after read budget is available.

- **Generated harness artifacts need a regenerate-and-diff gate, not memory.** `npm run build` refreshed `public/data/harness.json` from 582 to 600 durable concepts and updated the category-link and tier-2-doc counts, proving the live harness page can lag behind the repo unless generated artifacts are rebuilt and committed. How to apply: add a CI gate that runs the deterministic generators (`glossary`, `harness`, `creators`) and fails on a diff, the same way doc path references and count claims already fail.

- **Glossary index centering is a page-level concern, separate from concept-hub centering.** Concept hubs already used `.hub-page { margin-inline: auto }`, but the glossary index rendered as a plain `.stack` inside the 1180px shell and therefore hugged the left. How to apply: list or directory pages that are meant to read as editorial surfaces need their own centered page class, not just centered detail pages.

- **Do not run `npm test` and `npm run build` concurrently in this repo.** The build runs deterministic generators that rewrite JSON under `public/data/`, while tests such as the style scan read those same artifacts. A parallel local run produced a transient `Unexpected end of JSON input` while a file was being rewritten; a sequential rerun passed. How to apply: build and test are both required gates, but run them one after the other unless the generators use atomic writes or the tests read an isolated snapshot.

## Session 17, 2026-06-05: Nav cut to a few primary pills plus a right-anchored Menu, the broken search removed, and the interior pages centered

- **A floating panel that overlaps page content must be OPAQUE, never a translucent glass.** The header command palette's dropdown used `color-mix(in oklab, var(--surface) 86%, transparent)` plus a backdrop blur, so on the Explore page (which has its own search box and concept rows) the panel and the page behind it bled together into an unreadable overlap (Eric's broken_search.png). The frosted treatment is fine for the thin sticky HEADER, where content merely scrolls under a narrow bar, but a content-overlapping menu or dialog needs a solid `var(--surface)` fill. The new `.app-menu-panel` is opaque for exactly this reason. The palette was removed from the header for now (the `src/components/CommandPalette.jsx` file remains, unmounted) rather than patched, because Eric judged the search not worth keeping live in its broken state. How to apply: reach for translucency only when nothing readable sits behind the element; default dropdowns, popovers, and modals to an opaque surface.

- **An interior reading page must center its column (`margin-inline: auto`), not merely cap its width.** Methodology, About, Harness, Review, and Learn each set a `maxWidth` (60 to 82ch) with no auto margin, so the column pinned to the LEFT of the 1180px shell with a wide empty gutter on the right, which reads as broken (Eric flagged it directly against the worldthought.com room pages). ## Session 6, 2026-06-02: The rotation plane was the wrong instrument; replace it with trend heat boards

- **A four-quadrant rotation plane (Mansfield, the prior art behind RRG) needs a persistent universe sampled continuously, and this corpus is the opposite, so the plane showed clamp artifacts, not rotation.** Pulled from the real `*_week.json`: 12 of 16 techniques had `ratio` pinned to exactly the `[55, 145]` clamp wall; three concepts that co-occur in the same items (`prompt-engineering`, `workflow-orchestration`, `agent-wrapper`) were byte-identical dots stacked on each other; and almost every trail had the same shape (appear at center, jump to a corner, slide down the wall), which is an exponential-decay kernel over a publish-date histogram, not a topic rotating. How to apply: before committing to a visualization, sanity-check it against the REAL artifact values, not the math in the abstract; a metric that looks principled (Mansfield normalization) can be degenerate on the data regime you actually have (sparse, bursty, recency-dominated, churning universe).

- **The replacement ranks by ABSOLUTE windowed growth (`delta`), and the metric had to be chosen against real numbers because the obvious normalized version also saturated.** First attempt was a symmetric normalized score `(recent - prior) / (recent + prior)` on the decayed level. The opinion board now shows genuine decliners at the bottom (`AI agents` delta -8.6, `test-time compute` delta -10.7: large but cooling), exactly the heating-versus-cooling read the plane never gave. How to apply: when "trending" is the goal on sparse data, prefer an absolute growth measure for the ranking (it degrades gracefully and never divides by a near-empty base); reserve any normalized ratio for display direction, where its saturation is harmless.

- **"Commits not getting pushed" was root-caused to there being no git remote and no upstream ever configured.** With no `origin`, `git push` had nothing to push to: it either errored or no-op'd, and the surrounding tooling reported success anyway, so finished work sat committed-only and the deployed tree silently diverged from local. The standing fix: push a new branch with `git push -u` to set the upstream, and ALWAYS verify after pushing that local HEAD equals the remote. `git status` must show `## main...origin/main` with nothing ahead (or run `git rev-parse HEAD` against `git rev-parse origin/main`). Treat a push as unverified until the local and remote SHAs match; never trust a "pushed" report on its own. The remote now exists at `https://github.com/aigamma/ai-firehose.com` (public, MIT).

- **A hermetic CI gate is cheap insurance against exactly this kind of drift.** Added `.github/workflows/ci.yml`: on push and PR to `main`, one ubuntu job runs `npm install` (no committed lockfile, so install not `ci`), `npm test`, then `npm run build`. It uses `permissions: contents: read`, a cancel-in-progress concurrency group, and no secrets. The `mcp/` smoke test is deliberately excluded: it does live network I/O and `mcp/` may be untracked on a clean clone, so it would be non-hermetic. Verified locally that both `npm test` (28 passing) and `npm run build` succeed with no env vars before wiring the workflow.

- **`docs/Commands` and `package.json` scripts must agree: an unbacked command is a doc lie.** The docs reference `npm run network`, but no such script exists, and `worker/pipeline/network.mjs` cannot back one: it only `export`s pure functions (no `main()`, no CLI entry), and `computeSpectrums` needs a live Voyage `embedFn`. All the orchestration (load the store, embed, write every dependent artifact) lives in `run.mjs`, so the network rebuild is a stage of `npm run ingest`, not a standalone command. Did NOT add a fake `network` script; the discrepancy is the doc's to fix. How to apply: before documenting a command, confirm a script (or a runnable entrypoint) actually backs it; grep `package.json` scripts for any `npm run X` a doc mentions.

- **The MCP package must not depend on this repo.** `mcp/package.json` listed `"ai-firehose": "file:.."`, which ties the published npm package back to the monorepo, breaking both `npx ai-firehose-mcp` for end users and `npm publish`, and defeating the portable, no-repo design (the server wraps the public API and needs only `@modelcontextprotocol/sdk`). Removed it. How to apply: a package meant to be published standalone must declare only real, published dependencies; a `file:` dependency is a local-dev convenience that does not survive publication.

## Session 2, 2026-06-02: Overnight hardening (split the heavy JSON payloads)

- **Precomputed artifacts are a served payload, not a scratch dump: ship only what a page renders.** The glossary `index.json` had grown to 1.08 MB because it carried the full hub (definition, neighbors with scores, axis_positions, top_items) for all 427 concepts, and both the glossary list and every `/technique/:slug` hub downloaded the whole thing to use a slice of it. `spectrums.json` was 591 KB because each axis embedded its 1024-dim `axis_vector` (needed only by a not-yet-built server projection) plus a raw `position` next to the `position_normalized` the UI actually reads. `neighbors.json` (287 KB) was served but fetched by nothing (already denormalized into the hubs). Why it matters: this is a static site, so every committed artifact is downloaded by real visitors, and the heaviest sat on the two most data-rich pages. How to apply: split per-entity hubs into `glossary/c/<id>.json` fetched on demand, keep a slim `index.json` (`id, label, kind, attention, aliases, def_snippet`) for list and search, strip any field no consumer reads, and park server-only data (axis vectors) outside `public/`. Result: glossary list 1.08 MB to 88 KB, each hub about 4 KB, spectrums 591 to 398 KB, neighbors removed. The fix touched both the generator (`run.mjs`) and the already-committed artifacts (`worker/pipeline/slim_artifacts.mjs`, an idempotent one-time migration), because fixing only the generator would leave the live tree heavy until the next ingest.

- **Audit who actually fetches an artifact before adding fields to it.** Grep each artifact name across `src/` and `netlify/`. A field only the precompute step or a future feature needs does not belong in the browser payload. Caveat learned here: the dedicated ripgrep-based search tool respects `.gitignore`, so it silently skips the built `dist/` bundle and even returned empty for strings that exist in tracked source during this session; a raw recursive `grep` is the authoritative cross-check before deleting or reshaping a served file.

- **The MAX backstop and per-source limits are working: a real run produced 258 items across 6 of 7 sources.** YouTube 104, Hacker News 32, arXiv 30, blog 34, GitHub 25, Hugging Face 33; Reddit still 0 from this datacenter IP (expected, works from Fly or residential). 427 concepts after AI-grown resolution. How to apply: this is a healthy reference distribution to sanity-check future runs against; a sudden collapse in one source's count is an adapter regression, not a quiet day.

- **Do not prematurely declare "complete" and drop into standby: a fresh-eyes audit of an already-"finished" site found eight commits of genuine work.** Re-reading the actual artifacts instead of trusting a prior "done" claim surfaced a 1 MB glossary payload, two spec'd-but-missing hub fields (momentum, first_seen), a silent concept-loss bug, and a stale Methodology paragraph. Why it matters: the loop's value is real improvement, and idle "standby" with tight pulses is exactly the theater the global harness rules forbid. How to apply: when tempted to stand by, instead audit one concrete dimension end to end (payload sizes, referential integrity, spec vs implementation, a house rule) and measure it. Either you find real work or you bank a verifiable green check, both better than a pulse. When the safe, locally-verifiable backlog is genuinely exhausted, keep the loop alive on a long keepalive interval rather than a 250s pulse, and say so plainly.

- **A prompt instruction is not enforcement: enforce house rules in code.** Every model prompt says "no em dashes", yet models emit them anyway, and the displayed prose (classifier summaries, glossary definitions) had no guard. Added `stripEmDashes` (`worker/lib/text.mjs`) applied in `classify.mjs` and `define.mjs` so model output is sanitized before it enters the corpus. Why it matters: the committed definitions happened to be clean, but that was luck, and summaries live only in Pinecone where the artifact test cannot see them. How to apply: when a house rule governs model output, add a deterministic post-processor at the point the output enters the system, and unit-test the processor; treat the prompt instruction as a hint, not a guarantee. En dashes (numeric ranges) and verbatim source titles are deliberately left alone.

- **The classifier can emit `concepts` as a comma-joined string, not an array.** One stored item carried a string; `canonicalizeConcepts` iterates with for..of, so it walked the string character by character and dropped every char as junk, silently losing that item's concepts (no crash, because `remap` returns an array). How to apply: concepts are now coerced at the single chokepoint (`toConceptList` in `concepts.mjs`). If a future change reads raw `item.concepts` elsewhere, coerce there too, and watch for the same shape drift in `entities`.

- **An accumulating store is durable state, so it must be committed, not gitignored.** `worker/.cache/items.json` (the rolling-quarter corpus, the substrate for every rebuild) was gitignored as if it were a cache, but the worker clones the repo fresh each run with no persisted volume. So neither the Fly worker nor a CI runner would accumulate the quarter: each run starts empty, the corpus collapses to the current feed snapshot, and the Pinecone reconcile then deletes everything else. OPERATIONS.md's recovery recipe even assumed the raw items were staged. Fix: commit `items.json` (retention bounds it to ~0.5 MB) and add it plus `feed.xml` to the publish commit-back; the genuinely regenerable caches (classify, definitions, axis_vectors) stay ignored. How to apply: separate durable state from cache by lifecycle, not by directory. If losing a file silently degrades the system across runs, it is state and must persist.

- **An accumulating store keyed by content hash silently duplicates re-edited source items.** A YouTube video re-titled between runs (creators A/B test titles constantly) gets a new content hash, hence a new deterministic id, and `store[id] = it` added it alongside its stale prior version; retention only prunes by `published_at`, which a re-title does not change. The store had two entries for one video ("Anthropic is about to IPO" and "GPT-5.6 about to DROP", same URL). Over a quarter of daily runs this compounds, double-counting a popular video's concepts in the rotation and cluttering the RSS feed. Fix: `collapseStore` (`worker/pipeline/store.mjs`) keeps one entry per `(source, source_id)`, preferring this run's freshly classified version, else the most recently published. How to apply: any store keyed by a content hash needs a collapse step on the stable source key; the within-run dedup (which keys on `source:source_id`) does not catch cross-run re-edits.

- **Batch embedding and rerank APIs tag each result with its input `index`; reorder by it, never trust array order.** `voyage.mjs embed()` pushed embeddings in response order, which would silently misalign a whole batch with its inputs if the API ever reordered, corrupting every similarity, cluster, neighbor, and spectrum, the most catastrophic-if-wrong path in the system. Fixed with `orderEmbeddings` (place each at `d.index`). How to apply: the same holds for any OpenAI-style batch endpoint; Voyage rerank already maps via `r.index` in `retrieve.mjs`, so that path was already safe.

- **Audit the docs against the implementation, not only the code against itself.** CLAUDE.md's core "Classification Without Guessing" and "AI-Grown Taxonomy" contracts described a `_proposed.json` human-approval staging gate, an OpenAI dual-check, and registry-driven thresholds (0.86/0.78), none of which the resolver actually did: `concepts.mjs` hardcoded 0.9/0.8, never read the registry, and auto-entered every new concept with no staging. The keystone protocol treats docs that lie as a process failure equal to code that lies, so this is not cosmetic. Resolution this session: wired the registry `TAXONOMY` thresholds into the resolver so the single source of truth is real, then rewrote CLAUDE.md, the registry comment, and `docs/INGESTION.md` to describe the implemented algorithm and to mark the approval gate and dual-check as intended-but-not-implemented. How to apply: when a contract names a file or constant (`_proposed.json`, `mergeThreshold`), grep for it; an unreferenced name is either dead config or a lie. Prefer implement-to-match when it is safe; choose doc-to-match when the code is sound and re-running to validate a behavior change is not (here: no keys, and a threshold change re-clusters the whole taxonomy).

## Session 1, 2026-06-01: Founding (planning and scaffold)

The decisions and principles that shaped the project, captured so a fresh agent inherits them without re-deriving them.

- **The core is robotlogic.org's non-chat embedding machinery, not a chatbot.** The most advanced RAG work in the portfolio (constellation, centroids, clusters, concept spectrums, influence graph, neighbors) is all non-conversational. Why: that machinery organizes and visualizes a corpus, which is exactly what this site needs. How to apply: do not add a chatbot. The only live model surface is semantic search via `netlify/functions/retrieve.mjs`.

- **Retention is a feature: the corpus self-expires at one quarter.** Why: in the fastest-growing industry, stale signal is noise, and a bounded corpus keeps Pinecone and Voyage costs flat forever. How to apply: the prune stage deletes by `published_at` past `RETENTION_DAYS` (registry). The four horizons are nested windows inside that single quarter; Quarter is the hard outer bound.

- **The taxonomy is AI-grown and fitted loosely, never a fixed tag list.** Why: a fast field coins new concepts constantly, and exact-string tagging both misses them and fragments near-duplicates. How to apply: the model discovers and names new concepts, then every candidate is reconciled to the existing taxonomy by voyage-3 cosine (`TAXONOMY` thresholds in registry). Bind at or above 0.86, review in the ambiguous band, create-new below 0.78 into `_proposed.json` for in-repo approval.

- **Three per-kind rotation boards, and the rotating entity differs per kind.** Techniques rotate by concept, Tools by named product or repo, Opinions by discourse theme. Why: Eric chose three boards plus one unified constellation. How to apply: one shared attention-and-rotation computation, three entity sources. (Superseded Session 4: the unified constellation was removed; the three rotation boards remain.)

- **YouTube is the primary source, coded from scratch.** Channel RSS for new-upload detection, yt-dlp for metadata and captions, Whisper for caption-less videos (the exact civil transcript pattern). How to apply: Apify is documented only as a paid fallback if RSS or yt-dlp is blocked. Favorite teachers get high `source_authority_weight`.

- **Public, no auth; curation happens in-repo.** Why: simplest, matches the sibling sites, and the human-author stop (approving new glossary concepts) fits a file or PR review.
